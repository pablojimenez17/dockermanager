# OrbitCloud DockerManager Architecture Documentation

## Overview
This document provides a comprehensive overview of the architecture of the OrbitCloud DockerManager repository as it pertains to the configuration files and components that facilitate our Docker deployment. Each section delves into the details of specific files, configurations, and interactions.

## Components
The architecture consists of the following key components:
1. **docker-compose.yml**
2. **Networks**
3. **Volumes**
4. **Dockerfiles**
5. **HAProxy Configuration**
6. **Edge Firewall Setup**

---

## 1. docker-compose.yml
The **docker-compose.yml** file serves as the backbone of the Docker orchestrated environment. Below is a detailed explanation:
### Example Structure:
```yaml
version: '3'

services:
  app:
    build:
      context: ./app
      dockerfile: Dockerfile
    networks:
      - app-network
    volumes:
      - app-data:/data

networks:
  app-network:

volumes:
  app-data:
```
### Explanation:
- **version**: Defines the version of the Docker Compose file format.
- **services**: Specifies the services (containers) to be deployed, with each service potentially having unique settings (build context, networking configurations, etc.).
- **networks**: Lists the networks created to facilitate communication between services.
- **volumes**: Defines persistent data storage used by the containers.

## 2. Networks
In this architecture, networking is crucial for inter-service communication. The example above illustrates how we define a custom network called `app-network` to connect our `app` service:
### Key Features:
- **Isolation**: Each service can only access other services in the same network.
- **Inter-service Communication**: Services on the same network can communicate using their container names as hostnames.

### Diagram:
```plaintext
+-------------------+          +-------------------+
|       app        |          |       db         |
|   (Container)    | <------> |   (Container)     |
+-------------------+          +-------------------+
        |                             |
        +-----------------------------+
                 |  app-network
```  

## 3. Volumes
Volumes are utilized to persist data generated by and used by Docker containers. In the example of the `app` service, we map the volume `app-data` to `/data` in the container:
### Benefits:
- **Data Persistence**: Data is not lost when containers are stopped or removed.
- **Shared Data**: Volumes can be shared among multiple containers.

### Diagram:
```plaintext
+-------------------+
|      Volume       |
|    app-data       |
+-------------------+
         |
         +---------------> /data (app)
```  

## 4. Dockerfiles
Dockerfiles are scripts that contain a series of instructions to create Docker images. An example Dockerfile used to build the `app` service might look like:
### Example of a Dockerfile:
```Dockerfile
FROM node:14
WORKDIR /app
COPY . .
RUN npm install  
CMD [ "npm", "start" ]
```
### Explanation:
- **FROM**: Specifies the base image to build upon.
- **WORKDIR**: Sets the working directory inside the container.
- **COPY**: Copies local files to the container.
- **RUN**: Executes commands to install dependencies before the container runs.

## 5. HAProxy Configuration
HAProxy is used for load balancing and proxying requests to Docker containers:
### Example HAProxy Configuration:
```plaintext
frontend http_front
    bind *:80
    default_backend http_back

backend http_back
    server app1 app:80 check
    server app2 app:81 check
```
### Explanation:
- **frontend**: Defines the entry point where it listens for incoming requests.
- **backend**: Specifies the servers to route the traffic to, facilitating load balancing.

## 6. Edge Firewall Setup
The edge firewall is essential for securing access to the Docker environment:
### Key Points:
- **Rules**: Explicitly allowing necessary ports (e.g., 80 for HTTP).
- **Monitoring**: Continuously checking for unauthorized access attempts.
